Best Practices

These are some best practices that we can adhere to for our project. Best practices are not limited to this list and any suggesstions to improve the process will be added to the list.

  1. Branch:
    • Create branch names that reflect the Userstory or defect. <Userstory | Defect Number>_<Description> For Ex: B-46972_AOS_Save_And_Fetch
    • Create separate branch for each stories/defects.
    • Let us ensure that we are not stepping on each other branches.
    • Delete the remote branch as soon as a merge request is merged to master/develop branch. This ensures that no one is using outdated branches.
    • Before you delete your branch, please ensure that feature branch is approved and merged to master
    • Delete your local feature branch as well.
    • Always watch for GitLab/Slack for new MRs and it status.
    • Rebase your branch if any previous MRs were approved and merged.
  2. Commits:
    • Like branch names, prefix commit message with task id. <Task ID>_<Task Description> For Ex: TK-101860_Create ingestion logic to push data to MainLine table.
    • Commit message with taks ID helps to relate commits to story tasks.
    • You can commit any number of times with the same commit message as long as it is relevant. This ensures that you are not missing your local changes at any given point of time.
    • You can squash one or more relevant commits just in case if you want to make it as single commit.
  3. Approval:
    • We will have minimum 2 approvers and maximum of 3 approvers per merge request. It's always good to have extra pairs of eyes.
    • To approve merge requests
      1. Please checkout/switch to the merge request branch.
      2. Do a unit test to ensure that any functionality isn't broken.
      3. In case of client side MR, please ensure that we don't see any JS errors on the console.
      4. Have a detailed review at the code. This is to ensure that erroneous code getting into codebase.
      5. Please take MRs as high priority and provide your approval ASAP.
  4. Review:

    • Reviews and comments aren't to point fingers on others.

    • Reviews make our codebase stronger, error free and fool proof.
    • Approach review comments positively.
    • By reviewing others work, you are getting a different dimension of coding constructs. The more you review the more you learn.
    • Provide code snippets in comments wherever you feel comments can't express your idea.
    • Make necessary changes as per the review comments, check-in the changes, respond to comment and close it.
    • Reviews need not be once per MR. In case, the comment is misunderstood or further code optimization can be done, the whole review process shall continue.
